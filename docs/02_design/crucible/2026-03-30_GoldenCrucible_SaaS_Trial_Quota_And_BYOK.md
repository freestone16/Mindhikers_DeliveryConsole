# 2026-03-30 GoldenCrucible SaaS 试用额度与通用 BYOK 方案

## 1. 目标

本方案只服务 `GoldenCrucible SaaS` 当前上线阶段，不进入 `SSE` 主线。

目标：

- 让新注册用户可以直接体验黄金坩埚
- 控制平台侧 `Kimi 原厂 k2.5` 成本与滥用风险
- 在免费额度耗尽后，引导用户切换到自己的模型接入
- 保持实现简单、可审计、可逐步扩展

当前原则：

- 默认平台模型：`Kimi 原厂 k2.5`
- 平台免费额度：有限额度试用
- 超限后：进入 `BYOK`
- `BYOK` 不强绑定 Kimi，允许用户自行填写 `baseUrl + apiKey`

---

## 2. 核心决策

### 2.1 免费试用规则

每个注册用户默认拥有：

- `3` 个对话
- 每个对话最多 `10` 轮

定义：

- `1 个对话` = 创建一个新的 `conversationId`
- `1 轮` = 用户发出一次有效问题，并成功拿到一次模型回答

补充规则：

- 用户在已有对话内继续聊，只消耗该对话的剩余轮数
- 某个对话达到 `10` 轮后，不允许继续在该对话内调用平台模型
- 当 `3` 个对话全部耗尽后，平台免费额度结束

### 2.2 超限后的产品行为

当用户超出免费额度时：

- 不封账号
- 不删除历史对话
- 不影响查看历史内容
- 只阻止继续调用平台模型
- 页面弹出 `BYOK` 引导

### 2.3 BYOK 口径

`BYOK` 不再限定为 Kimi。

用户最少需要填写：

- `baseUrl`
- `apiKey`

建议同时支持：

- `model`
- `providerLabel`（可选，仅用于 UI 展示）

默认引导文案仍可优先推荐：

- `Kimi 原厂 k2.5`
- `baseUrl = https://api.moonshot.cn/v1`
- `model = kimi-k2.5`

但产品能力上不写死为 Kimi。

## 3. 安全边界

### 3.1 平台 Key

平台侧 `Kimi` key 用于免费试用额度。

要求：

- 只保存在服务端 secrets
- 绝不下发到前端
- 绝不写进浏览器 localStorage
- 前端只能请求自己的后端接口

### 3.2 用户 BYOK

第一阶段建议采用“轻量 BYOK”：

- 用户在前端填写 `baseUrl / apiKey / model`
- 数据只对当前登录用户生效
- 服务端按用户维度保存加密后的 BYOK 配置
- 实际调用时由服务端代用户请求对应模型接口

说明：

- 这不是“key 永不进云端”的最强 BYOK
- 但它能兼顾：
  - 多端登录
  - 历史会话续聊
  - 正式 SaaS 可用性

如果后续要升级到“真 BYOK”，再单开二期：

- 前端本地保存 key
- 服务端不保存用户 key
- 代价是用户换设备或换浏览器后需要重新填写

### 3.3 风控

免费试用阶段至少增加：

- 用户维度额度统计
- IP 维度请求频率限制
- 注册冷却或邮箱验证
- 单轮输入长度上限
- 单用户并发会话上限

## 4. 数据口径

建议新增两类状态：

### 4.1 用户额度状态

- `trialConversationLimit = 3`
- `trialTurnLimitPerConversation = 10`
- `trialConversationsUsed`
- `trialTurnsUsedInCurrentConversation`
- `trialStatus`

`trialStatus` 建议枚举：

- `active`
- `conversation_exhausted`
- `quota_exhausted`
- `byok_enabled`

### 4.2 用户模型接入状态

- `mode = platform | byok`
- `byokBaseUrl`
- `byokModel`
- `byokProviderLabel`
- `byokConfigured`
- `byokLastValidatedAt`

`apiKey` 不回传前端明文。

---

## 5. 产品行为细则

### 5.1 新用户

- 注册完成后直接可进入产品
- 默认走平台 `Kimi 原厂 k2.5`
- 首次创建对话时扣除第 `1` 个免费会话名额

### 5.2 免费对话过程中

- 每成功完成一轮后扣减该对话剩余轮数
- 前端显示：
  - 当前对话已用轮数
  - 剩余免费对话数

### 5.3 免费额度耗尽

触发条件：

- 已创建 `3` 个免费对话，且都达到上限
  或
- 当前创建新对话时，免费会话额度已用尽

用户体验：

- 弹窗或侧板提示“免费额度已用完”
- 不再直接报错
- 给出 `BYOK` 配置入口

### 5.4 BYOK 配置成功后

- 当前用户切换到 `mode = byok`
- 后续对话优先走用户自己的 `baseUrl + apiKey + model`
- 历史会话可继续聊
- 平台免费额度不再继续扣减

---

## 6. UI 范围

本轮只做 SaaS 自己需要的最小入口。

### 6.1 右上角入口

恢复右上角 `LLM 配置` 入口，但改成 SaaS 口径：

- 默认展示文本模型配置
- 不展示图生
- 不展示视生
- 不展示与当前 SaaS 无关的模块配置

### 6.2 超限引导面板

用户免费额度耗尽时显示：

- 当前额度已用完
- 推荐填写自己的模型接入信息
- 表单字段：
  - `Base URL`
  - `API Key`
  - `Model`

可选字段：

- `服务商名称`

---

## 7. 后端改动范围

### 7.1 需要新增

- 试用额度检查逻辑
- 平台模式 / BYOK 模式路由逻辑
- 用户维度 BYOK 配置存储
- BYOK 连通性测试接口

### 7.2 需要收口

- 当前 `llm-config` 页面是“大而全”设计，需要裁成 SaaS 专用版本
- 当前平台默认配置仍有 `deepseek / siliconflow` 历史口径，需要统一改为：
  - 推荐默认：`Kimi 原厂 k2.5`

---

## 8. 明确不做

本轮先不做：

- 多模型 marketplace
- 团队共享额度
- 按 token 精细计费
- 用户自定义多套 key 轮换
- 真正意义上的“前端本地无云存储 BYOK”

---

## 9. 任务拆分

### T1. 免费额度与路由判定

目标：

- 为每个注册用户加上 `3 对话 * 10 轮` 试用额度
- 每次发起对话前判断走：
  - 平台模型
  - 用户 BYOK
  - 超限拦截

完成标志：

- 后端能正确判断 `platform / blocked / byok`

### T2. 超限引导与前端状态提示

目标：

- 在免费额度即将耗尽和已耗尽时给出明确提示
- 已耗尽时自动引导到 BYOK 配置入口

完成标志：

- 用户不会看到生硬报错，而是看到清晰下一步

### T3. SaaS 专用 LLM 配置页

目标：

- 恢复右上角入口
- 只保留文本模型相关配置
- 默认推荐 `Kimi 原厂 k2.5`
- 增加通用 `BYOK` 配置字段：
  - `baseUrl`
  - `apiKey`
  - `model`

完成标志：

- 用户能看到并保存自己的 BYOK 配置

### T4. 验收与风控补线

目标：

- 验证平台试用模式可用
- 验证超限拦截可用
- 验证 BYOK 模式可用
- 增加基础 rate limit 与审计日志

完成标志：

- 能完成一轮完整 smoke：
  - 新用户免费使用
  - 免费额度耗尽
  - 配置 BYOK
  - 继续对话成功

---

## 10. 当前建议执行顺序

建议直接按以下顺序推进：

1. 先做 `T1 免费额度与路由判定`
2. 再做 `T2 超限引导`
3. 接着做 `T3 SaaS 专用 LLM 配置页`
4. 最后做 `T4 验收与风控`

这条线足够独立，适合直接在 `GoldenCrucible-SaaS` 当前分支内继续完成。
