# Distribution Terminal 前端改造参考稿（Mixpost 风格，兼容现有后端）

> 日期：2026-03-22
> 状态：Implemented (Phase 1)
> 参考对象：Mixpost 的前端信息架构与交互层级
> 明确边界：**只参考前端组织方式，不迁移 Laravel / Vue / Inertia 技术栈**

---

## 1. 设计目标

当前 Distribution Terminal 的后端能力已经基本成型：

- `Magic Fill` 真数据源已接入
- Queue 已支持 SSE
- `X / 微信公众号` 已具备一期执行语义

但前端仍存在三个问题：

1. **页面过散**：`Publish Composer` 仍像三个并列表单，不像一个统一的“发帖工作台”
2. **重心不够聚焦**：平台选择、文案编辑、提交策略没有形成明显主次
3. **Queue 像日志面板**：有信息，但缺少“运营工作台”的扫描效率

因此本稿目标是：

1. 参考 Mixpost，把 Distribution 前端收成更成熟的发布体验
2. **后端尽量不动**，继续复用当前 API 与类型模型
3. 先做一期 IA / 视觉层重构，不急着上日历、媒体库、分析面板

---

## 2. 设计原则

### 2.1 借鉴什么

借鉴 Mixpost 的：

- **单贴中心化 Composer**
- **平台版本（Platform Versions）**
- **右侧发布控制栏**
- **队列卡片的层级化结果呈现**

### 2.2 不借鉴什么

不借鉴 Mixpost 的：

- Laravel / Vue / Inertia 技术组织
- Workspace / Team / Analytics / Media Library 大体量模块
- 完整 Calendar-first 产品形态

### 2.3 MHSDC 必须保留的特点

- `projectId` 驱动
- `Magic Fill` 来源展示
- `sourceFiles / platformOverrides / deliveryMode`
- `06_Distribution/` 项目内落盘协议
- `Accounts Hub / Publish Composer / Queue` 三页式宿主结构

## 3. 新的信息架构

## 3.1 当前结构问题

当前 `Publish Composer`：

- 左边选视频
- 中间填全局文案
- 右边勾平台

这对“功能打通”有效，但不符合成熟发布器的认知路径。

用户真实心智更像：

1. 我要发哪条内容？
2. 发到哪些平台？
3. 每个平台有没有特殊文案？
4. 是立即、排队，还是定时？

## 3.2 目标结构

新结构建议改成：

```text
┌──────────────────────────────────────────────────────────────────────┐
│ Header：Publish Composer / 项目 / Script / 快速状态                  │
├───────────────────────────────────────┬──────────────────────────────┤
│ Main Composer                         │ Right Rail                   │
│                                       │                              │
│ A. Content Identity                   │ 1. Platform Summary          │
│    - 选视频/视频卡                    │    - 已选平台 chips          │
│    - Magic Fill 来源                  │    - 兼容性提示              │
│                                       │                              │
│ B. Core Post                          │ 2. Publish Mode              │
│    - 标题                             │    - Queue                   │
│    - 正文                             │    - Schedule                │
│    - Tags                             │    - Timezone                │
│                                       │                              │
│ C. Platform Versions                  │ 3. Submit CTA                │
│    - X 卡片（可展开）                 │    - 确认起飞                │
│    - 微信公众号卡片（可展开）         │    - 风险说明                │
│    - 其他平台卡片（折叠）             │                              │
│                                       │ 4. Validation Panel          │
│ D. Source + Validation                │    - 缺视频                  │
│    - sourceFiles                      │    - 缺正文                  │
│    - warnings                         │    - 未选平台                │
└───────────────────────────────────────┴──────────────────────────────┘
```

这会比当前“三栏并排”更像一个完整工作流。

## 4.1 Publish Composer（建议主改）

### 目标版线框

```text
┌────────────────────────────────────────────────────────────────────────────┐
│ Publish Composer                                     [CSET-SP3] [Script]  │
│ 跨平台提稿机 - 一键分发全网                                              │
├─────────────────────────────────────────────────────┬──────────────────────┤
│ 内容主体区                                          │ 发布控制区           │
│                                                     │                      │
│ [视频卡片] 16:9  CSET-SP3-S1-minimal-test.mp4       │ 已选平台             │
│ [视频卡片] 16:9  CSET-SP3-S1-v2.mp4                 │ [X] [公众号]         │
│                                                     │                      │
│ [从真实产物自动装填]                                │ 兼容性               │
│ 来源：YouTube_Submission_v6.txt · selected video    │ Shorts / 抖音 / 视频号
│                                                     │ 仅 9:16              │
│ 标题                                                │                      │

│ └─────────────────────────────────────────────────┘ │ ( ) 定时发布         │
│                                                     │ [时间] [时区]        │
│ 正文                                                │                      │
│ ┌─────────────────────────────────────────────────┐ │ 校验提示             │
│ │ ...                                             │ │ - 已选择视频         │
│ │ ...                                             │ │ - 已命中文案源       │

│                                                     │                      │
│ Tags                                                │ [确认起飞 (2)]       │
│ ┌─────────────────────────────────────────────────┐ │                      │
│ │ Blue Lobster, OpenClaw ...                     │ │                      │
│ └─────────────────────────────────────────────────┘ │                      │
│                                                     │                      │
│ 平台定制                                            │                      │
│ ┌ X / Twitter ───────────────────────────────────┐ │                      │
│ │ 自定义标题                                      │ │                      │
│ │ 自定义 Tags                                     │ │                      │
│ └─────────────────────────────────────────────────┘ │                      │
│ ┌ 微信公众号 ─────────────────────────────────────┐ │                      │
│ │ 自定义标题                                      │ │                      │
│ │ 摘要预览（只读/自动生成）                       │ │                      │
│ └─────────────────────────────────────────────────┘ │                      │
└─────────────────────────────────────────────────────┴──────────────────────┘
```

### 为什么这样改

1. **主内容区只处理“这条内容本身”**
   - 视频、标题、正文、Tags 放在一起
   - 用户会感觉自己在编辑“一条发布内容”，不是在填三个表

2. **平台差异收敛到 Platform Versions**
   - 把当前 checkbox 下零散出现的自定义标题 / tags，改成“平台卡片”
   - 这就是 Mixpost 风格里最值得借鉴的一点

3. **发布控制区固定在右侧**
   - 已选平台
   - 兼容性提示
   - 立即/定时
   - 最终 CTA
   - 这样提交决策永远稳定，不会随着正文长度被挤下去

4. **Magic Fill 成为“来源能力”，而不是单独功能块**
   - 现在按钮还在，但它不应该抢主视觉
   - 更像“用真实项目产物增强这条内容”

## 4.2 Queue（建议次改）

当前 Queue 已可用，但太偏“事件日志 + 结果列表”。

建议改成：

```text
┌──────────────────────────────────────────────────────────────────────┐
│ Distribution Queue                           [实时同步中] [刷新]     │
│ 发布工作台                                                            │
├───────────────┬──────────────────────────────────────────────────────┤
│ 左侧筛选      │ 主列表                                                │
│               │                                                      │
│ 状态          │ ┌ 任务卡片 ────────────────────────────────────────┐ │
│ - 全部        │ │ 已发布      3月22日 10:18                        │ │
│ - 待执行      │ │ MIN103 connectors smoke 20260322                 │ │
│ - 执行中      │ │ twitter · artifact_ready                         │ │
│ - 已发布      │ │ wechat_mp · draft_ready                          │ │
│ - 待重试      │ │ 最近事件：任务执行完成                            │ │
│               │ │ [查看物料] [重试] [删除]                          │ │
│ 平台          │ └──────────────────────────────────────────────────┘ │
│ - X           │                                                      │
│ - YouTube     │ ┌ 任务卡片 ────────────────────────────────────────┐ │
│ - 微信公众号   │ │ 待重试 ...                                       │ │
│               │ └──────────────────────────────────────────────────┘ │
└───────────────┴──────────────────────────────────────────────────────┘
```

### Queue 的核心改动点

1. **把“最近事件”从顶部主角降为辅助区**
   - 实时状态很重要
   - 但不应压过任务列表本身

2. **任务卡片要更像“结果单元”**
   - 展示 `deliveryMode`
   - 若有 `artifactPath`，给“查看物料”按钮
   - 若有 `url`，给“打开链接”

3. **增加轻量筛选**
   - 状态筛选
   - 平台筛选
   - 不需要上复杂搜索，先提升扫描效率

## 5. 与当前后端的映射关系

这套前端改造的好处是：**几乎不要求后端重构**。

## 5.1 Composer 映射

当前已有：

- `assets.title`
- `assets.textDraft`
- `assets.tags`
- `assets.sourceFiles`
- `assets.platformOverrides`

因此前端只需要把 UI 重组，不需要重写 payload 模型。

## 5.2 Queue 映射

当前已有：

- `task.status`
- `task.latestEvent`
- `task.results[platform].message`
- `task.results[platform].deliveryMode`
- `task.results[platform].artifactPath`
- `task.results[platform].url`

因此 Queue 也只需要增强呈现，不需要新增核心状态字段。

## 6. 推荐改造顺序

## Phase A：低风险重组（推荐先做）

只改前端布局和组件层级：

1. `PublishComposer`
   - 左主区 + 右侧栏
   - 平台区改为 card/accordion
   - 提交按钮固定在右侧

2. `DistributionQueue`
   - 顶部实时条缩小
   - 列表卡片层级增强
   - 增加轻量筛选

这一步基本不动 API。

## Phase B：交互补强

1. 平台卡片支持折叠/展开
2. 右侧栏显示“缺少什么还不能发”
3. Queue 卡片增加“查看物料”入口

## Phase C：再决定要不要借 Calendar

这一阶段再看是否需要：

- 周视图 / 月视图
- Queue 槽位
- 推荐发布时间

一期不建议先做。

---

## 7. 我建议的最终取舍

### 7.1 值得直接做

我建议直接推进：

1. **Composer 改成单贴中心化**
2. **平台定制改成卡片式 versions**
3. **右侧固定发布控制栏**
4. **Queue 增加筛选 + artifact/button 呈现**

### 7.2 暂时不要做

1. 不上日历
2. 不上媒体库
3. 不上 analytics
4. 不改后端主模型

---

## 8. 一句话判断

如果要参考 Mixpost，最正确的姿势不是“把它搬过来”，而是：

> **用 Mixpost 的前端心智，重组我们现在已经打通的 Distribution 能力。**

这样我们能同时得到：

- 更成熟的前端体验
- 更低的实现风险
- 对现有后端最小扰动
