# Distribution Terminal · 一期 PRD（v1）

> 本文是分发终端一期产品决策的唯一事实来源（SSOT）。
> 头脑风暴稿（`docs/brainstorms/2026-04-26-distribution-terminal-requirements.md`）已被取代，仅作为决策溯源参考。
> 实施细节、文件级改造、测试计划由后续 `ce-plan` 产出，不在本文档范围。

---

## 0. 一句话定义

**让独立创作者打开一个项目，就能看见这个项目要发到 4 个平台的全部发布卡 — 每张卡的文案预填好、有问题置灰、没问题一键发，发完区分「真发了」还是「只到草稿箱」。**

这不是一个"多平台发布工具"，是一个"项目分发台"——视角从「素材」转成「项目」。

---

## 1. 用户与节奏

| 维度 | 设定 |
|---|---|
| 用户 | 独立创作者（一期就一个：老卢） |
| 发布频率 | 每周 3-5 次 |
| 一次发布的范围 | 围绕**单个项目**，向 4 个平台一次性发完 |
| 工具熟悉度 | 高（创作者本人写代码用工具） |
| 风控敏感度 | 中等（一期不需要多账号矩阵反检测） |

**这个画像决定了**：
- ✅ 不做多人协作、权限、日历视图
- ✅ 不做账号矩阵 / IP 池 / 反检测调度
- ✅ UI 信息密度可以高，不必一切都"小白友好"
- ✅ 失败重试默认即可，不需要复杂策略
- ❌ **必须**做的：草稿态明示、账号健康前置、风控开关可关

## 2. 核心产品决策（一图定方向）

| # | 决策 | 拒绝的反方案 | 决策理由 |
|---|---|---|---|
| **D1** | **按项目维度组织**，素材组分块 | 按平台维度组织 / 按素材库浏览 | 创作者真实工作流是"一个项目产出 → 多平台分发"。素材库是开发视角，项目台才是用户视角 |
| **D2** | 卡片数量 = **素材数 × 适配平台数**，不固定 | 固定 4 张卡 | 一个项目可能有 1 篇长文 + 1 部成片 = 4 卡；也可能 2 篇长文 + 2 部成片 = 8 卡。卡数随素材线性增长 |
| **D3** | **草稿态独立呈现**，永不混淆为"已发布" | 把草稿态当 success / 把不能直发的平台砍掉 | 公众号 API 仅支持创建草稿，B站社区方案多为草稿/审核流。如不区分，用户会以为发了实则没推送给粉丝 |
| **D4** | 字段**两层**：通用层 + 平台高级（可展开） | 一上来铺所有平台所有字段 | 标题/正文/标签覆盖 80% 用例，平台高级字段是 20% 场景，藏起来降低认知负荷 |
| **D5** | 风控延时**默认开 + 可关** | 默认关（裸发） / 默认开且不可关 | 独立创作者风险低但有概率被检测；可关满足"我想立刻看效果"的合理需求 |
| **D6** | 重试 3 次但**按错误码分类** | 一刀切重试 / 不重试 | 401/内容违规重试无意义只会刷屏，429 应延迟重试，5xx 应立即重试。分类是对的，写死间隔是错的 |
| **D7** | 一期收敛到 **4 个平台**：X / 公众号 / YouTube / B站 | 8 个全做（含抖音/视频号/Shorts） | 这 4 个的内容形态和授权机制差异最大，覆盖完即覆盖了所有难点。其余平台的代码占位保留，UI 隐藏 |
| **D8** | 账号健康**永远在场**（顶部健康条） | 单独的"账号管理"页 | 账号红灯应立刻让对应卡置灰；健康条是分发台的健康检查仪 |
| **D9** | **不做视频组装流水线** | 一并做 ffmpeg 切片 / Remotion 叠加 | 这是 Delivery Console 上游的事；分发终端只消费已经成片的素材 |
| **D10** | 旧 PublishComposer **保留为旁路** | 删除 / 全替换 | 自由选片自由发是合法二级场景。主战场是项目分发台，PublishComposer 标 Beta 留作"散件入口" |

## 5. 平台字段定义

> 仅一期需要的字段。完整字段映射在 ce-plan 阶段细化。

### 5.1 X / Twitter
- **通用**：标题、正文、标签
- **平台高级**：reply settings（谁可回复 / followers only / mentioned only）、community URL、made_with_ai 标识、paid partnership 标识
- **发布产物**：单推 或 线程（自动按字符切分）
- **直发模式**：✅ 已发布

### 5.2 微信公众号
- **通用**：标题（建议 ≤20 字）、正文（无字数上限）、标签
- **平台高级**：摘要（≤120 字）、作者、原文链接、封面图（→ thumb_media_id）、评论开关、赞赏开关
- **发布产物**：单图文草稿（`/cgi-bin/draft/add`）
- **直发模式**：📝 草稿态（用户必须登录 mp.weixin.qq.com 手动群发）

### 5.3 YouTube
- **通用**：标题、描述、标签
- **平台高级**：visibility（public/unlisted/private）、madeForKids、category、license、comments 开关、缩略图
- **发布产物**：长视频（一期不做 Shorts）
- **直发模式**：✅ 已发布（支持 publishAt 定时）

### 5.4 B站
- **通用**：标题、简介、标签
- **平台高级**：copyright（原创/转载）、tid（分区）、封面、禁止转载、充电开关、杜比音效、Hi-Res、定时发布（dtime）
- **发布产物**：稿件（投稿待审核）
- **直发模式**：📝 草稿态（投稿后等 B站审核 + 用户后台确认）

## 6. 验收标准

完成下列每一项即认为一期可交付。

- [ ] 选定项目后，分发台正确扫描出 N 张卡（N = 素材组数 × 平台数），ready/置灰状态正确
- [ ] 顶部健康条 4 平台状态正确反映 `/auth/status` 返回值，红灯平台对应卡置灰
- [ ] 能进入单卡编辑页，修改通用层和平台高级字段，保存到 task.assets
- [ ] 草稿态平台（公众号 / B站）：编辑页有横幅说明、卡片有虚线橙徽章、队列页有「打开后台」按钮
- [ ] 直发平台（X / YouTube）：发布成功后卡片显示真实 URL 可点击
- [ ] 多选 + 立即发布能一次性入队多张卡
- [ ] 多选 + 定时发布按分钟精度生效，到点自动执行
- [ ] 队列页 SSE 实时反映状态变化，状态筛选 / 平台筛选 / 项目筛选可用
- [ ] 自动重试按错误码分类：401 不重试、429 递增 1/5/15min、5xx 立即重试，3 次失败后停
- [ ] 风控延时开关默认开，可关；关闭后任务立即执行
- [ ] X / YouTube / 公众号 / B站 4 个 connector 全部能跑通最小成功路径

## 7. 范围边界

### 7.1 In Scope（一期做）
- 项目分发台（按项目维度的 4+ 卡视图，含素材组分块）
- 4 平台 connector：X/YouTube 真直发，公众号/B站 草稿态
- 编辑页两层字段、平台高级设置可展开
- 立即 / 定时（分钟精度）双发布模式
- 队列 SSE 实时监控，草稿态/已发布/失败的视觉规约
- 自动重试 3 次（错误码分类）+ 手动重试
- 风控延时开关（默认开）
- 单人账号绑定（OAuth / API / Cookie 各按平台特点）
- 账号健康条（顶部常驻）

### 7.2 Out of Scope（一期不做）
- ❌ 视频组装流水线（ffmpeg 切片 / Remotion 叠加）—— 上游 Delivery Console 的事
- ❌ YouTube Shorts / 抖音 / 视频号 / 微博 等其它 4 个平台
- ❌ 多人协作 / 权限管理 / 账号矩阵
- ❌ 日历视图 / 媒体库 / 数据分析面板 / AI 推荐发布时间
- ❌ 跨项目批量操作（只支持单项目内多卡批量）
- ❌ 内容合规审查 / 违禁词检测

## 8. 依赖与假设

### 8.1 上游产出约定

| 产物 | 来源模块 | 项目内路径（约定） |
|---|---|---|
| 长文（图文） | 写作大师 | `02_Script/*.md`（含封面图同名 PNG） |
| 长视频成片 | 影视导演 P4 | `04_Video/*.mp4`（一期主成片） |
| 平台文案 | 市场大师 | `05_Marketing/<platform>.md`（如 `05_Marketing/youtube.md`） |

如某项目目录结构与约定不符，分发台扫描后该素材组以「未 ready」呈现，不阻塞其它素材组。

### 8.2 平台 API 假设

- **X API v2**：用户已申请 free/basic tier，能 POST tweets（每月 1500 条限额够用）
- **YouTube Data API v3**：用户已开通项目 + OAuth 已配（项目内已有现成实现）
- **微信公众号**：用户已申请订阅号/服务号，有 AppID/AppSecret，能调 `/cgi-bin/draft/add`
- **B站**：一期采用 cookie 注入方式（biliup-rs 子进程 或 直接调 mock 接口）—— 选型由 ce-plan 决定

### 8.3 技术栈

- 前端：React + Vite（沿用现有项目）
- 后端：Node.js + Express（沿用现有 server/）
- 队列：当前为内存 + JSON 持久化（一期不引入 BullMQ，待二期评估）
- 实时推送：SSE（已就绪）

## 9. 待决议问题

### 9.1 必须 ce-plan 阶段解决（影响实施）

1. **B站 connector 选型**：biliup-rs 子进程桥接 vs playwright cookie 注入 vs 直接 HTTP 请求加 cookie。各方案的稳定性 / 部署成本 / 维护成本对比
2. **公众号封面图链路**：用户上传图 → 调微信素材库 API 换 thumb_media_id → 写入草稿。这条链是分两步还是合一步？
3. **重试间隔的具体配置**：429 递增 1/5/15min 是否合理？需不需要做成可配置？
4. **定时发布的时区**：所有定时统一存 UTC + 用户本地时区显示？还是直接存本地时区？涉及 YouTube/B站 自带的定时字段映射

### 9.2 可推到二期

- X 的 `made_with_ai` / `paid_partnership` 是否需要企业级 API 权限（付费层级）
- B站 tid 分区选择器：硬编码常用分区表 vs 调 B站接口动态拉
- 公众号"群发后通知用户"的回调机制（一期可手动）
