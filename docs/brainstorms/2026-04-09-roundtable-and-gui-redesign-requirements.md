# GoldenCrucible 圆桌讨论引擎 + GUI 风格对齐

## Development Philosophy

**奥卡姆剃刀 | 有价值 | 简单 | 强壮 | 易维护**

核心竞争力 = 思考哲学 + 数据协议 + 流程编排。UI 是低耦合的渲染层，功能不依赖特定布局。

## Problem Frame

GoldenCrucible 当前的深聊模式（老卢+老张追问用户）缺少前置发散环节。用户直接带着未经碰撞的议题进入深聊，起点质量受限于个人认知边界。

需要一个**圆桌讨论层**，由多位哲人/专家辩论，暴露议题中的核心冲突，提炼出 2-3 个经过碰撞检验的命题。用户可选择其中 1 个进入深聊，也可以只做圆桌发散，或直接跳过圆桌进入深聊。

### 与 ljg-roundtable 的关系与差异化

参考了 [lijigang/ljg-skill-roundtable](https://github.com/lijigang/ljg-skill-roundtable) 的辩证对话编排模式，但在以下维度做本质区分：

| 维度 | ljg-roundtable | GoldenCrucible Roundtable |
|------|---------------|--------------------------|
| 人格来源 | LLM 通识知识，每次临时生成 | **结构化人格档案**（PersonaProfile JSON），低耦合可插拔 |
| 人格深度 | MBTI + 一句话立场 | **一期：价值立场+修辞风格两层**；二期扩展至四层复合模型 |
| 一致性保障 | 无 | 前置对比约束（prompt 结构保证） + 轮内记忆锚定 |
| 用户角色 | 手动控制每轮 | **有干预权的导演**：旁观 + 控制节奏 + 轻量参与（投/？） |
| 可扩展性 | 固定于 Claude Code CLI | 功能层与渲染层分离，GUI/CLI 共用同一引擎 |
| 输出 | org-mode 笔记 | **Spike 列表 + 讨论全文**，Spike 可导入深聊或持久化保留 |
| 下游 | 无 | 命题→深聊→论文→内容改编，完整管道 |

## Requirements

**人格档案协议（PersonaProfile Contract）**

- R1. 定义 `PersonaProfile` 接口作为所有人格数据的统一契约。圆桌引擎**只认合同，不认生产者** — 无论是手工编写、LLM 生成、nuwa 式蒸馏、还是未来任何引擎产出，只要符合接口即可消费
- R2. PersonaProfile JSON 采用**必填+可选**分层设计：
  - **必填（一期）**：`identity`（姓名、时代、领域、一句话定位）、`values`（Haidt MFT 6 foundations 量化）、`rhetoric`（Ethos/Pathos/Logos 偏好 + 文体指纹）
  - **可选扩展**：`mental_models`（心智模型列表，参考 nuwa-skill 格式）、`heuristics`（决策启发式）、`expression_dna`（表达DNA：句式偏好、高频词、节奏特征）、`personality_traits`（Big Five facets）、`argumentation`（Toulmin 偏好）、`honest_boundaries`（诚实边界：此人不会说什么）
  - **元数据**：`source`（手工/llm_generated/engine_distilled）、`version`、`created_at`
  - 引擎读取时：有扩展字段就用来增强 prompt，没有就只用必填字段，绝不崩溃
- R3. PersonaProfile 以独立 JSON 文件存储，一个人物一个文件，存放于约定目录（如 `personas/`），运行时按需加载
- R4. 预制 7 位人物档案：苏格拉底、尼采、王阳明、汉娜·阿伦特、查理·芒格、费曼、赫伯特·西蒙。一期用 LLM 基于学术共识生成必填字段，人工审核
- R5. PersonaProfile 加载机制必须支持热插拔 — 新增 JSON 文件即自动可用，无需改代码。二期可接入任意生产者（nuwa 式蒸馏引擎、用户自定义引擎等），只要输出符合接口契约

**命题锐化（Proposition Sharpening）**

- R6. 用户输入模糊议题后，系统用 1 轮快速追问将其锐化为一个有张力的可辩论命题，再进入圆桌
- R7. 锐化步骤极轻量：不是圆桌，不是多轮对话，只是系统用 2-3 个针对性问题帮用户收窄议题范围、明确立场边界
- R8. 如果用户输入的已经是明确命题（系统判断），可跳过锐化直接进入圆桌或深聊

**圆桌讨论引擎（用户导演模式）**

- R9. 圆桌讨论接收一个命题（来自锐化步骤、用户手动输入、或未来的信息采集模块），哲人自主辩论
- R10. 系统基于议题从 PersonaProfile 库中自动选择 3-5 位参与者，优先覆盖多元立场和最大化思想张力
- R11. 讨论编排参考 ljg-roundtable 模式：主持人引导 → 参与者发言（回应前文，非独白）→ 主持人综合提取争议焦点
- R12. 主持人每轮生成 ASCII 思维框架可视化，暴露讨论结构
- R13. 讨论核心原则：求真 > 和谐，挖深不铺广，每轮聚焦一条最深裂缝

**用户导演指令**

- R14. 节奏控制：`可`（继续下一轮）、`止`（结束讨论）、`深入此节`（锁定当前争议深挖）、`换人`（替换或引入新参与者）
- R15. 轻量干预：`投`（向圆桌抛入一个观点或事实，哲人必须回应后继续辩论，用户不下场）、`？`（点名追问某位哲人一个具体问题）
- R16. 用户角色定位 = **有干预权的导演**，不是参与者也不是纯观众

**Spike 提取与输出**

- R17. 讨论结束时产出两份内容：a) Spike 列表 — 2-3 个结构化命题（争议命题、支持方/反对方、核心论据摘要、张力度评分）b) 讨论全文 — 圆桌完整对话记录
- R18. 用户从 Spike 列表中选择 1 个导入深聊，其余命题持久化保留，后续可随时再选

**三条路径支持**

- R19. 圆桌讨论和深聊均为独立入口，用户可自选路径：a) 完整管道（圆桌→选Spike→深聊）b) 纯圆桌（只做发散碰撞，收获 Spike 列表）c) 直接深聊（跳过圆桌，带命题直接进入老卢老张追问）
- R20. 完整管道作为推荐路径，但三条路径平权，不强制顺序

**圆桌→深聊桥接**

- R21. 选中的 Spike 自动转化为深聊的 `InputCard`，携带圆桌阶段的争议上下文（不传原始全文，只传结构化摘要）
- R22. 深聊阶段的老卢和老张能感知该命题在圆桌中的碰撞历史，追问时可引用哲人的关键论点
- R23. 未选中的 Spike 持久化存储，在侧边栏或命题库中可见，用户可随时选择任一 Spike 开启新一轮深聊

**角色差异性保障（前置对比约束）**

- R24. 通过 prompt 结构保证角色差异性，而非后置 embedding 校验：a) 每位哲人的 system prompt 中注入与其他参与者的对比锚点（"你必须与 X 形成张力"）b) 每轮生成时，将前一位哲人的核心论点作为"必须回应/挑战的靶子"传入下一位的 prompt
- R25. 每位参与者维护独立的轮内记忆摘要，用于锚定其核心立场防止人格漂移
- R26. 人格引擎的深层机制（独立上下文、一致性评估、参数敏感度验证等）作为**独立调研课题**，产出单独的需求说明书供专人调研。参考 [nuwa-skill](https://github.com/alchaincyf/nuwa-skill) 的蒸馏框架但不照搬，核心是确保 PersonaProfile 接口的开放性

**信息采集接口预留**

- R27. 圆桌引擎的初命题入口定义为标准接口（接收 topic + optional context），不绑定特定来源
- R28. 预留"灵魂文件"筛选接口：接收候选议题列表 + 用户灵魂档案，输出筛选后的初命题。本期不实现，只定义接口契约

**GUI 风格微调（低耦合，低优先级）**

- R29. 视觉风格向 Claude Code 对齐：深色主题、等宽字体、配色方案可直接复刻 Claude Code
- R30. 布局默认竖版 2-3 联（侧边栏 + 对话流 + 按需展开的黑板区），但布局机制支持用户调整
- R31. 当前 header 功能迁移至侧边栏，释放主内容区垂直空间
- R32. 侧边栏入口：圆桌讨论（导演模式）、深聊（参与模式）、命题库（历史 Spike）
- R33. 三条路径在入口处清晰呈现，用户一目了然可选

**SSE 流式适配**

- R34. 圆桌讨论复用现有 SSE 流式基础设施，扩展 `CrucibleTurnEvent` 支持多发言人轮次
- R35. 圆桌阶段的讨论过程通过 SSE 实时推送到前端，用户旁观哲人辩论并可随时发出导演指令

## Success Criteria

- SC1. 圆桌运行 3-5 轮后，产出的 2-3 个 Spike 有实质性差异且有真实思想张力
- SC2. 至少 1 个 Spike 是用户在输入初命题时未预料到的角度
- SC3. 任意两位圆桌参与者的发言在风格和立场上可明显区分（通过前置对比约束实现，无需后置校验）
- SC4. 新增一个 PersonaProfile JSON 文件后，无需改动任何代码即可在下次圆桌中使用
- SC5. 选中的 Spike 导入深聊后，老卢和老张的追问能自然引用圆桌碰撞中的关键论点
- SC6. 三条路径均可独立走通，不存在强制依赖
- SC7. 圆桌引擎的所有功能不依赖特定 UI 布局，可在 CLI 上下文中同等运行

## Scope Boundaries

- **不做**：人格提取引擎（单独产出需求说明书供专人调研）
- **不做**：人格模型第3-4层（Big Five / Toulmin，待调研验证 prompt 敏感度后再决定）
- **不做**：后置 cosine similarity 校验（用前置对比约束替代）
- **不做**：信息采集模块（本期只预留接口）
- **不做**：灵魂文件筛选逻辑（本期只定义接口契约）
- **不做**：CLI 形态（本期只做 GUI 渲染，但引擎层必须与 UI 解耦）
- **不做**：阶段3内容改编（后续独立 PRD）
- **不做**：大规模 UI 重构（一期只做 Claude Code 风格对齐 + 侧边栏迁移）
- **不做**：Fine-tuning 或 LoRA（纯 prompt engineering 驱动人格模拟）

## Key Decisions

- **三条路径平权，不强制管道**：圆桌→深聊是推荐路径，但用户可单独使用圆桌或直接深聊。理由：已有成熟想法的用户不应被迫看一场不需要的辩论
- **用户 = 有干预权的导演**：不是纯观众（会走神），不是参与者（精力留给深聊），而是可以 `投` 观点、`？` 追问的导演。理由：保持参与感的同时不抢哲人的主场
- **命题锐化前置**：圆桌启动前 1 轮快速追问，把模糊感觉锐化为可辩论命题。理由：避免哲人浪费 1-2 轮帮用户定义问题
- **一期两层人格模型，二期验证后扩展**：价值立场 + 修辞风格先行，Big Five + Toulmin 预留接口但不填充。理由：纯 prompt engineering 对 40-50 参数的敏感度未经验证，避免过度设计
- **前置对比约束替代后置校验**：用 prompt 结构（对比锚点 + 靶子传递）保证差异性，不做 embedding 计算。理由：延迟更低、治本不治标、工程更简单
- **人格引擎独立调研**：独立上下文、一致性评估、参数敏感度等深层机制，产出单独需求说明书。理由：这是一个需要专人深入调研的课题，不应在本 PRD 中草率决策
- **功能 > UI**：核心竞争力在思考哲学、数据协议和流程编排。引擎层与渲染层严格分离
- **Spike 是圆桌→深聊的硬接口**：结构化命题，不传原始全文，只传结构化摘要
- **GUI 微调而非重设计**：复刻 Claude Code 配色/字体/布局风格即可
