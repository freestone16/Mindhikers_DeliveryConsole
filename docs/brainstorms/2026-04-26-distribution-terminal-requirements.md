# Distribution Terminal (分发终端) 产品需求文档

## Problem Frame

Distribution Terminal 是 MindHikers 创作者工具矩阵的**分发编排层**，承接 Delivery Console 产出的内容资产，完成最后一公里的多平台发布。

**当前问题**：
- 技术文档（组装流水线、三圈层分发矩阵、开源融合策略）非常完整，但都在回答「怎么做」
- 缺少产品视角的「做什么、为谁做、做到什么程度算好」
- 前端已有 PublishComposer 和 DistributionQueue，但体验偏「功能打通」，缺少成熟发布器的认知路径

**目标**：定义一期最小可用产品，让创作者能**按项目维度浏览素材 → 逐条确认/编辑文案 → 选择平台 → 发布 → 看结果**。

---

## Requirements

### R1. 项目维度浏览

- **R1.1** 用户在分发终端选定一个项目后，系统扫描项目约定目录，自动生成 4 条发布记录（X / 微信公众号 / YouTube / B站）
- **R1.2** 每条记录显示平台图标、ready 状态、内容类型（图文/视频）
- **R1.3** Ready 判定规则：
  - X / 公众号：项目约定目录下存在写作大师产出的长文文件 → ready
  - YouTube / B站：视频目录有视频文件 **且** 市场大师文案目录有文案方案 → ready
- **R1.4** 未 ready 的条目置灰显示，不可勾选/发布，但可点进去查看缺失什么

### R2. 逐条编辑与确认

- **R2.1** 用户点击某条发布记录，进入该平台的编辑界面
- **R2.2** 编辑界面分两层：
  - **通用层**（始终显示）：标题、正文、标签
  - **平台高级设置**（可展开）：各平台特有的字段
- **R2.3** 各平台高级字段定义（参考 GitHub 高星开源项目）：
  - **X / Twitter**：reply settings（谁可回复）、community URL、made with AI、paid partnership
  - **微信公众号**：摘要（≤120字）、作者、封面图（thumb_media_id）、评论开关、贴图模式（newspic，5-20张图）
  - **YouTube**：保持现有字段结构（visibility、madeForKids、category、license、comments、thumbnail）
  - **B站**：copyright（原创/转载）、分区（tid）、封面、禁止转载、充电开关、杜比音效、Hi-Res、定时发布（dtime）
- **R2.4** 系统根据上游模块产出自动预填通用字段（标题、正文、标签），用户可修改
- **R2.5** 编辑后的文案仅在当前发布记录生效，不影响上游原始文件

### R3. 平台选择与发布

- **R3.1** 用户在项目视图看到 4 条发布记录，每条右侧有复选框，可多选
- **R3.2** 仅 ready 状态的条目可被勾选
- **R3.3** 用户勾选后，底部出现「发布」按钮，显示已选平台数量
- **R3.4** 发布策略二选一：
  - **立即发布**：点击后立即加入队列执行
  - **定时发布**：按分钟精度设定未来时间（参考 YouTube Studio），到点自动执行
- **R3.5** 发布前需二次确认弹窗，展示：发布平台列表、定时时间（如有）、风控提示

### R4. 队列与状态监控

- **R4.1** 发布任务进入队列后，用户在 Queue 页面实时查看状态（SSE 推送）
- **R4.2** 任务卡片展示：项目名、平台图标、状态（排队中/执行中/成功/失败/待重试）、最近事件
- **R4.3** 状态筛选：按状态（全部/待执行/执行中/已发布/待重试）、按平台筛选
- **R4.4** 成功的任务显示平台返回的 remoteId / url（如有），提供「打开链接」按钮
- **R4.5** 失败的任务显示错误日志，提供「重试」和「删除」按钮

### R5. 失败重试

- **R5.1** 每个平台任务自动重试最多 3 次，每次重试间隔递增（如 1min / 5min / 15min）
- **R5.2** 3 次重试均失败后，任务标记为「失败」，推送通知给用户
- **R5.3** 用户可在 Queue 中手动触发重试，手动重试同样遵循 3 次上限

### R6. 账号授权管理（预留接口）

- **R6.1** 一期实现单人账号绑定，支持 X OAuth、YouTube OAuth、微信公众号 API、B站 Cookie/Token
- **R6.2** 账号状态显示：已绑定（绿灯）/ 未绑定（红灯）/ 过期告警（黄灯）
- **R6.3** 架构上预留多人账号池扩展接口，但不实现多用户权限管理

## Success Criteria

- [ ] 选定项目后，4 条发布记录正确显示 ready/置灰状态
- [ ] 能逐条进入编辑，修改标题/正文/标签，平台高级字段可展开
- [ ] 勾选多条记录后，能一次性发布到多个平台
- [ ] 定时发布按分钟精度设定，到点自动执行
- [ ] Queue 实时显示任务状态，成功有链接，失败有日志和重试按钮
- [ ] 自动重试 3 次失败后通知用户
- [ ] 4 个核心平台（X/公众号/YouTube/B站）至少打通 2 个（一期目标 X + 公众号 + YouTube）

## Scope Boundaries

### In Scope (一期)
- 按项目维度的 4 平台发布记录浏览与 ready 判定
- 逐条编辑确认 + 批量发布
- 立即发布 + 分钟级定时发布
- Queue 状态监控 + SSE 实时推送
- 自动重试 3 次 + 手动重试
- 单人账号绑定（X OAuth、YouTube OAuth、公众号 API、B站 Token）

### Out of Scope (一期不做)
- 视频组装流水线（SD-301 的 ffmpeg 裁剪/Remotion 叠加）
- 国内平台高强度逆向（抖音、视频号）
- 多人协作与权限管理
- 日历视图 / 媒体库 / 数据分析面板
- AI 推荐最佳发布时间
- 跨项目批量操作

## Key Decisions

| 决策 | 理由 |
|------|------|
| 按项目维度组织，而非素材库浏览 | 更符合创作者 workflow：一个项目产出 = 多平台分发 |
| 4 平台条目同时呈现，未 ready 置灰 | 让用户一眼看到全貌，知道还缺什么 |
| 平台字段分层：通用 + 高级展开 | 降低认知负荷，高级用户可深入 |
| 定时发布按分钟精度 | 参考 YouTube Studio，满足精细化运营需求 |
| 自动重试 3 次 | 平衡自动化与可控性，避免无限重试导致封号风险 |
| Postiz 参考设计（不抄代码） | AGPL 许可证限制，仅借鉴字段定义和 DTO 模式 |
| biliup-rs / jiji262-wechat-publisher 直接参考 | MIT 许可证，字段定义可直接借用 |

## Dependencies / Assumptions

- **上游模块约定**：写作大师产出长文到项目约定目录（如 `02_Script/`）；市场大师产出文案到约定目录（如 `05_Marketing/`）；视频文件放到视频目录（如 `05_Shorts_Output/`）
- **平台 API 稳定性**：X API v2、YouTube Data API v3、微信公众号 API、biliup-rs 模拟接口在一期内保持稳定
- **账号授权**：用户已提前在各平台开通开发者账号并获取 API 凭据
- **技术栈**：前端 React + Vite，后端 Node.js + BullMQ 队列，SSE 实时推送
