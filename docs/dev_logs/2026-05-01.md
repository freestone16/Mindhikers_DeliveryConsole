# 2026-05-01 — Director Design System Target

## 概览

本日工作把 Director UI 从“已有暖纸面风格”推进为“可复用设计系统目标”。核心不是再做一轮散乱 UI 优化，而是先确立 `design.md / design.zh.md` 作为事实源，再让 PRD、实施计划、AGENTS、README、rules、testing 和 HANDOFF 全部围绕这套事实源收口。

一句话判断：

> 不要把 Director 完全 Claude Code 化，否则会失去影视创作工作台的温度；但应该吸收 Claude Code 的克制、状态透明、命令优先和审计感，让它从“好看的 UI 风格说明”变成“其他模块真能复用的产品设计准绳”。

## 背景

用户提出：能否把导演大师当前字体、布局、UI 总结到项目根目录 `design.md`，让其他模块借鉴。随后明确指出 Google Stitch / Google Labs 已有 `DESIGN.md` 方向，并询问如果以 Claude Code UI 为准绳，这版还有哪些优化点。

本轮形成结论：

1. Director 不应完全变成 Claude Code 式冷工具界面。
2. Director 应保留影视创作工作台的温度。
3. Director 应吸收 Claude Code 的：
   - 克制
   - 状态透明
   - 命令优先
   - 审计感
4. `design.md` 应先作为目标事实源落下，再请 CE 专家同步 PRD、实施计划和治理资产，最后再实施。

## 外部调研摘要

按用户要求上网确认了 `DESIGN.md` 来源和共识：

- Google Stitch / Google Labs 方向把 `DESIGN.md` 作为面向 AI 设计/编码代理的设计系统事实源。
- 常见结构包括 tokens、视觉方向、字体、颜色、组件、布局、响应式、Do/Don't、Agent Prompt Guide。
- 本项目选择保留小写 `design.md`，但结构兼容 Google Stitch 风格。
