## 1. 这份文档解决什么

3 月中旬的多轮讨论后，黄金坩埚出现了两套容易混淆的“第一阶段”口径：

1. **本地开发口径**
   - 指当前本地可运行、可演示、可继续扩展的黄金坩埚主链是否已经收口。
2. **产品上线口径**
   - 指黄金坩埚是否已经形成可以给朋友、合作方、投资人演示使用的线上产品版本。

如果不把这两套口径拆开，后续会持续混淆：

- `MIN-38` 到底该不该关闭
- SaaS 是不是还算“第一阶段”
- 圆桌多人讨论到底是不是第二阶段

因此本稿只做一件事：

**把本地阶段、产品阶段和下一阶段项目的关系一次性说清。**

---

## 2. 已确认的阶段定义

## 2.1 本地第一阶段：已完成

本地第一阶段的定义是：

**以当前本地黄金坩埚的实际完成状态为准，视为已经收口。**

它包含的核心结果是：

- `Socrates -> Bridge -> UI` 主链已经成立
- 黑板内容与右侧对话分离已经成立
- `dialogue + presentables + turn_log` 已进入真实运行时
- Soul registry / 默认双灵魂 / 最小 orchestrator 骨架已经落地
- 当前本地版本已经可以作为后续 SaaS 与 Phase 2 的稳定底盘

对应 Linear：

- `MIN-38` `Crucible Phase 1-dev`

结论：

**`MIN-38` 应按“本地第一阶段开发父任务已完成”收口。**

---

## 2.2 产品第一阶段：仍包含 SaaS 演示上线

虽然本地第一阶段已经完成，但从产品口径看：

**黄金坩埚真正的第一阶段仍然包括 SaaS 版本上线可用。**

这个 SaaS 阶段的目标不是大规模商用，而是：

- 给朋友演示
- 给合作方演示
- 给投资人演示
- 验证“深度对谈 -> 结构化产出”这条链路的外部感知价值

这一阶段的约束也已经明确：

- 服务器负载不会大
- 可以优先追求低复杂度、低运维成本
- 重点是稳定演示体验，不是追求重型 SaaS 功能完备

因此产品第一阶段的重点讨论应转向：

1. 最小 SaaS 形态到底保留哪些能力
2. 部署平台与技术栈如何选
3. 登录、会话存储、Skill 打包要做到什么程度
4. 如何用最小代价支撑 demo 级线上体验

---

## 2.3 第二阶段：圆桌多人讨论

圆桌多人讨论已明确归入：

**黄金坩埚第二阶段。**

补充决策（2026-03-19）：

- Roundtable 不再单独做本地先行版本
- 直接作为云端 `Phase 2` runtime 调试与演进
- 本地版继续只承担 `Phase 1-dev` 稳定底盘职责，避免重复开发

对应 Linear：

- Project: `Crucible-0 Phase 2`

它的定位不是继续收本地主链，也不是 SaaS 上线收尾，而是：

- 前置预热
- 冲突暴露
- spike 提取
- 让正式坩埚在更高密度的起点上开始

一句话：

**圆桌多人讨论不是 Phase 1 的尾巴，而是 Phase 2 的开场。**

---

## 3. 当前统一口径

截至 2026-03-19，统一口径如下：

1. `MIN-38`
   - 代表本地 Phase 1-dev
   - 现在可以收口
2. 产品层 Phase 1
   - 仍包含 SaaS demo 上线
   - 这是接下来重点讨论的方向
3. `Crucible-0 Phase 2`
   - 代表圆桌多人讨论阶段
   - 这是下一个小阶段的主要研发项目
   - 默认按云端 runtime 调试，不再单开本地 roundtable 版本

---

## 4. 后续讨论建议顺序

为了避免下一轮再次把阶段混在一起，建议按下面顺序推进：

1. 先以本稿为准，统一本地文档与 Linear 的阶段口径
2. 再集中讨论 SaaS Phase 1 的最小上线方案
3. 讨论完 SaaS 口径后，再拆 `Crucible-0 Phase 2` 的 roundtable MVP

这样可以保证：

- 本地主线不会被继续误判成“还没结束”
- SaaS 不会被误塞进圆桌阶段里
- 圆桌 Phase 2 能在清晰边界上启动
