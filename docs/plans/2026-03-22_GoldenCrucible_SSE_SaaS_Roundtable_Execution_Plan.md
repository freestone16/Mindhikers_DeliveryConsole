## 1. 本轮只聚焦的三项工作

本次对话后，黄金坩埚的当前执行面收束为三项：

1. **把黄金坩埚主链修订为 `HTTP + SSE` 版本并完成验收**
2. **完成 SaaS 版架构设计，明确环境需求，并完成部署上线**
3. **在黄金坩埚内开启 Roundtable 分支，同步完成本地版与 SaaS 版并最终合并**

## 2. 当前总判断

### 2.1 为什么要先做 SSE

黄金坩埚未来不是单一 web 应用，而是长期维护的多端产品：

1. SaaS Web
2. 本地 App
3. Agent 端
4. macOS / iOS / Android

在这种前提下，当前以 `Socket.IO` 为主的通信骨架需要让位于更通用的：

1. **命令层：HTTP**
2. **事件流层：SSE**

本轮不是为了“技术更潮”而做 SSE，而是为了：

1. 降低协议对单一前端运行时的绑定
2. 为 SaaS 部署扫清平台约束
3. 为未来多端和 agent 接入建立统一协议

### 2.2 本轮架构总原则

本轮所有实施都遵循这条原则：

**一个坩埚核心运行时，多个宿主端；一个统一协议层，多个终端入口。**

因此：

1. 不再把 Web 当前宿主视为唯一真身
2. 本地版与 SaaS 版共享同一套核心合同
3. Roundtable 分支也必须沿着同一套协议和运行时演进

## 3. 工作流 A：黄金坩埚 SSE 化与验收

## 3.1 目标

将黄金坩埚主链从当前偏 `Socket.IO` 的交互方式，收束为：

1. `HTTP Command`
2. `SSE Event Stream`

并完成可回归、可验收的主链验证。

## 3.2 范围

本工作流只覆盖黄金坩埚主链，不顺手改造其他模块。

纳入范围：

1. 坩埚主链请求入口重构
2. 坩埚主链事件流输出
3. 前端 SSE 事件消费与状态更新
4. 错误分层、超时、重连与终止语义
5. 主链验收与回归

暂不纳入范围：

1. Delivery / Distribution / Director / Shorts 全量 SSE 化
2. 复杂多人 presence
3. 协同编辑
4. 非坩埚宿主模块的实时链路重构

## 3.3 交付物

1. 坩埚统一命令入口
2. 坩埚统一 SSE 事件流
3. 前端事件消费器
4. SSE 事件类型清单
5. 验收清单与回归结果

## 3.4 验收标准

至少完成以下验收：

1. 用户发起新一轮对话时，请求可被标准 HTTP 命令接受
2. 首轮与续轮回复可通过 SSE 稳定流出
3. 中屏 `presentables` 能随事件正确更新
4. 错误事件不会被静默吞掉，前端能给出体面提示
5. 用户刷新或短时断连后，客户端可恢复到清晰状态
6. 坩埚主链不再依赖 `Socket.IO` 才能成立

## 3.5 实施边界

本轮目标不是“把仓库全切干净”，而是：

**先让黄金坩埚主链完成协议迁移并可验收。**

## 4. 工作流 B：SaaS 版架构设计、环境需求与部署上线

## 4.1 目标

在 SSE 主链成立后，完成黄金坩埚 SaaS 版的：

1. 运行时架构设计
2. 环境需求确认
3. 部署上线

## 4.2 SaaS 版的最小产品边界

本轮 SaaS 版只做一个可对外演示、可后续生长的在线宿主。

保留：

1. 坩埚对谈主链
2. 中屏黑板与结晶内容
3. 最小 session 保存与恢复
4. 最小 guest 或轻身份访问
5. 基础导出与错误提示

不保留：

1. Delivery / Distribution / Director / Shorts 宿主界面
2. 支付
3. 完整多租户
4. 复杂权限系统
5. 非核心增长组件

## 4.3 SaaS 版架构要求

SaaS 版必须满足：

1. 前端宿主与核心运行时清晰分层
2. 命令走 HTTP，流式结果走 SSE
3. 会话状态脱离本地工程目录语义
4. 不再依赖 `projectId / scriptPath / delivery_store.json` 才能跑通基础对谈
5. 能够为后续多端共用同一套协议层

## 4.4 环境需求

在正式部署前，需要用户提供或确认以下环境条件：

### 域名与入口

1. `mindhikers.com` 或现有 homepage 入口
2. `app.mindhikers.com` 作为坩埚 SaaS 入口
3. 可选的 staging 域名，用于部署前验收

### 运行时环境

1. 一个可持续运行的 Node.js 服务环境
2. 支持长连接响应与 SSE
3. 能配置环境变量
4. 能绑定自定义域名与 HTTPS

### 关键环境变量

1. LLM provider 相关 key
2. `APP_BASE_URL`
3. `API_BASE_URL`
4. `CORS_ORIGIN`
5. `SESSION_SECRET`
6. 会话存储连接串或等价持久化配置
7. 日志与错误监控配置

### 最小持久化

本轮至少需要：

1. 会话元数据持久化
2. 对话历史持久化
3. `presentables` / 核心产物最小持久化

本轮不要求一次性做完复杂数据平台，但不能继续把线上运行时建立在纯本地文件假设上。

## 4.5 上线验收

SaaS 版上线至少要满足：

1. 线上 app 域名可访问
2. 首页可跳转至 app
3. 用户能发起一轮对话并收到 SSE 返回
4. 中屏板书能正确出现
5. 基础错误提示、重试与重置成立
6. 线上环境变量与部署步骤有文档可复现

## 5. 工作流 C：Roundtable 分支、本地版与 SaaS 版同步、最终合并

## 5.1 目标

在黄金坩埚内部开启一个专门的 Roundtable 工作分支，使其：

1. 在独立分支中推进
2. 同步兼容本地版与 SaaS 版
3. 最终回合并到统一主线

## 5.2 原则

Roundtable 不允许形成“两套真身”：

1. 不做一套只给本地版用的 Roundtable
2. 不做一套只给 SaaS 版用的 Roundtable
3. 必须从一开始就围绕共享合同开发

## 5.3 分支策略

建议执行方式：

1. 创建专用 Roundtable 分支
2. 为该分支配置独立 worktree / 独立验证环境
3. Roundtable 只在该分支内推进，不直接侵入当前主线
4. 当本地版与 SaaS 版都完成对接验收后，再回合并

## 5.4 同步要求

Roundtable 分支推进过程中，必须同步满足：

1. 本地宿主可跑
2. SaaS 宿主可跑
3. 两边共享同一套协议与核心数据合同
4. 两边不分叉出不兼容的会话结构

## 5.5 合并标准

只有当以下条件都成立时，Roundtable 分支才允许合并：

1. 本地版可跑通 Roundtable 基础流程
2. SaaS 版可跑通 Roundtable 基础流程
3. 不破坏当前 Socrates 主链
4. 输入、事件、产物的核心 schema 保持一致
5. 有基本回归验证结果

---

## 6. 当前执行顺序

本轮建议按下面顺序推进：

1. **先完成 SSE 主链迁移与验收**
2. **再完成 SaaS 版架构收束、环境确认与部署上线**
3. **最后开启并推进 Roundtable 分支，完成本地版与 SaaS 版同步后再合并**

原因：

1. 没有 SSE，SaaS 版平台选择与多端协议都会继续摇摆
2. 没有 SaaS 主宿主，Roundtable 很容易继续只长在本地版上
3. Roundtable 必须建立在已经稳定的统一协议与宿主边界之上
